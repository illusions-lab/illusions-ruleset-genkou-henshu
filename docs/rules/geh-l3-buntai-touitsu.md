# geh-l3-buntai-touitsu — 文体（常体・敬体）混用の検出

> ⚠️ ステータス: L3（LLM支援）・未実装。SDK が L3 を支持した後に実装予定。本ドキュメントは仕様の先行定義です。

- **ruleId**: `geh-l3-buntai-touitsu`
- **nameJa**: 文体（常体・敬体）の混用検出
- **ガイドライン**: 原稿編集 第2版 (`genkou-henshu-2`)
- **レベル**: L3（LLM支援）
- **想定 severity**: warning
- **適用モード案**: official / academic / blog（novel は除外推奨）

## 書籍根拠

E 節「a） 表記の整理の原則」b) 表記整理の具体的事項 4 文体の整理。書籍は文体について常体（「……である」「……だ」調）と敬体（「……です」「……ます」調）の2種を示し、いずれかの文体で整理し混用を避けるよう求めている。さらに公用文については「である」調を原則とし、往復文書には「ます」調を用いるという使い分けも規定している。

## 動作原理

**入力**: 文書全体のテキスト（段落単位）  
**判断基準**:  
1. LLM が各文末表現（「〜だ」「〜である」「〜です」「〜ます」等）を分類し、文書全体の主体的な文体（常体 / 敬体）を決定する。  
2. 主体文体と異なる文末表現を持つ文を特定する。ただし、引用文・箇条書き・キャプション・注記など、文体が混在しても自然なコンテキストは除外する。  
3. 混用が連続して3文以上続く場合は「意図的な変更」と判断して除外し、孤立した逸脱のみを指摘する。

**出力**: 混用箇所の文と、文書の主体文体に合わせた修正案。

## なぜ L1/L2 で不可能か

- 常体・敬体の判定は文末形に加えて文脈が必要。「〜でした」は過去の叙述として敬体だが、一人称の思考述語では常体文中でも現れる。
- 助動詞「〜だった」「〜でした」の区別は形態素で取れるが、文書全体のどちらが「主体文体」かを確定するには複数文を横断した統計的・意味的判断が必要。
- 箇条書きや注・図キャプションなど構造的に文体が異なるブロックを自動除外するには、文書構造の理解が必要。

## 例

### positive（指摘なし）

```
本研究では新しい手法を提案した。実験の結果、精度が10%向上した。
提案手法は従来法と比較して計算コストが低い。
```
（全文が常体で統一されている）

---

```
本研究では新しい手法を提案します。実験の結果、精度が10%向上しました。
提案手法は従来法と比較して計算コストが低いです。
```
（全文が敬体で統一されている）

### negative（混用として指摘）

```
本研究では新しい手法を提案した。実験の結果、精度が10%向上しました。
提案手法は従来法と比較して計算コストが低い。
```
（2文目「向上しました」だけ敬体。主体文体の常体に揃えることを提案）

## 想定される偽陽性・運用上の注意

- **会話・引用**: 地の文が常体でも、台詞や引用内の敬体は正用。LLM は引用符・「」・""等で囲まれた範囲を除外すること。
- **意図的な文体転換**: 章や節の境界で文体を意図的に変える作品（一人称主観/三人称解説の切替など）は誤検出しやすい。severity は warning に留め、最終判断を著者・編集者に委ねる。
- **公用文**: 公告・通知等では本文が「である調」、宛先文が「ます調」を混在させる慣行があるため、文書種別を入力メタデータで指定できるようにする。
